# Task: Add edge provenance metadata

**ID:** `cw-edge-provenance-001`
**Type:** code
**Difficulty:** hard
**Rubric:** [`../rubrics/map_quality.md`](../rubrics/map_quality.md)

## Goal

Update `data/build_map_v2.py` so that generated edges include explicit provenance fields: `edge_type`, `confidence`, `explanation`, `created_at`, `source_script`. Update `map.html` so a user can inspect at least the `explanation` field for an edge (hover, click, or side panel).

## Context

The map currently has 2,687 edges between organizations. The edges are generated by keyword and metadata signals. Users have no way to know whether an edge represents a real-world relationship or a keyword artifact. This is the single weakest part of the network claim (see `FALSIFIERS.md` F-PROJ-1).

## Files

- `data/build_map_v2.py` (primary)
- `map.html` (display)
- Optional: a migration note in `data/migrations/` if the existing edge cache changes shape.

## Acceptance criteria

- [ ] Generated edges include `edge_type` (e.g. shared-framework-area, shared-source, shared-keyword, manual).
- [ ] Edges include `confidence` (low/medium/high or 0.0–1.0; document the choice).
- [ ] Edges include a human-readable `explanation` ("both organizations are tagged 'community land trust' from UK Charity Commission" or similar).
- [ ] Edges include `created_at` (ISO 8601) and `source_script` ("data/build_map_v2.py").
- [ ] The build script documents how confidence is assigned.
- [ ] `map.html` lets the user inspect at least the explanation for an edge.
- [ ] Existing map continues to render without breaking on edges that lack the new fields (backward-compatible default).

## Falsifier

Once provenance is in place, sample 50 edges and have a reviewer check whether the explanation matches a real-world relationship or only a keyword overlap. If <40% are real-world, the network framing in the README needs to be weakened.
