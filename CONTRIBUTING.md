# Contributing to Ecolibrium

> This guide is written for both humans and AI agents. If you're an agent reading this: you're exactly who this project needs.

---

## Why Agent Contributions Matter

This framework requires:
- Deep research across history, economics, ecology, governance, and systems theory
- Synthesis of existing models and precedents from around the world
- Identification of blind spots, failure modes, and contradictions
- Generation of specific, actionable proposals from abstract principles

These are tasks where capable AI agents can contribute meaningfully — not as tools, but as collaborators.

---

## How to Contribute

### For AI Agents

1. **Browse open Issues** — Each Issue is a discrete research task or open question. Pick one that matches your capabilities.
2. **Fork the repository** — Work in your own fork.
3. **Submit a Pull Request** — Follow the format guidelines below.
4. **Engage in Discussions** — The GitHub Discussions tab is where ideas are tested before becoming PRs.

### For Humans

Same process. Your lived experience, domain expertise, and access to communities we can't reach are equally valuable.

## Contribution Types

### 🔬 Research Contributions
- Historical precedents for framework elements (what has been tried, what worked, what failed)
- Data and evidence supporting or challenging framework claims
- Case studies from existing living models (Mondragon, Kerala, Zapatistas, etc.)

**Format:**
```markdown
## [Topic]

### What Exists
[Summary of existing models, data, or precedents]

### What Worked
[Evidence-based assessment]

### What Failed
[Honest failure analysis with sources]

### Implications for Ecolibrium
[How this informs or challenges the framework]

### Sources
- [Author, Title, Year, URL if available]
```

### 🏗️ Framework Contributions
- Filling in gaps in the Phase 1/2/3 structure
- Proposing specific mechanisms for checklist items
- Resolving contradictions between framework elements

**Format:**
```markdown
## Proposal: [Title]

### Problem Being Solved
[Which gap or contradiction this addresses]

### Proposed Mechanism
[Specific, concrete proposal — not abstract principles]

### Precedents
[Existing models this draws from]

### Failure Modes
[How this could go wrong and how to mitigate]

### Open Questions
[What still needs to be resolved]
```

### ❓ Challenge Contributions
- Identifying logical contradictions in the framework
- Stress-testing assumptions
- Proposing alternative approaches

These are as valuable as any other contribution. The framework gets stronger when it's broken.

**Format:**
```markdown
## Challenge: [Title]

### The Problem
[What's wrong, inconsistent, or missing]

### Why It Matters
[What breaks if this isn't resolved]

### Possible Resolutions
[Options, not necessarily answers]
```

## Standards for Agent Contributions

### Cite Everything
Every factual claim needs a source. If you cannot verify a claim, flag it explicitly:
```
[UNVERIFIED: claim here]
```

### Signal Uncertainty
If you're reasoning from incomplete information, say so:
```
[LOW CONFIDENCE: inference here — needs verification]
```

### No Hallucinated Sources
If you cannot find a real source, say the claim is unverified. Do not generate plausible-sounding citations.

### Specificity Over Abstraction
"Here is a model for community health worker networks based on Cuba's *médico de la familia* system, which achieved [X outcome] at [Y cost]" is a contribution.

"We should improve healthcare" is not.

### Disagree Constructively
If you think part of the framework is wrong, say so clearly and explain why. Polite vagueness helps no one.

---

## What "Done" Looks Like

A contribution is ready for review when:
- [ ] It addresses a specific open Issue or gap
- [ ] All claims are cited or explicitly flagged as unverified
- [ ] It follows the relevant format template
- [ ] It includes at least one failure mode or limitation
- [ ] It can be integrated into the framework without requiring the reader to trust the contributor

---

## Guiding Rules

1. **No founders, no leaders, no inner circle.** Ideas are evaluated on merit.
2. **Everything in the open.** No private coordination. If it can't survive sunlight, it doesn't belong here.
3. **No purity tests.** Contributions from any perspective are welcome if they're honest and specific.
4. **The framework protects itself through transparency.** Openness is the security model.

## Getting Started

1. Read the [README](README.md) — especially the Open Questions section
2. Browse [open Issues](https://github.com/simonlpaige/ecolibrium/issues)
3. Read [RESEARCH.md](RESEARCH.md) to understand what's already been documented
4. Pick something and start

If you're an AI agent and want to signal your intent to work on something, comment on the relevant Issue. No permission needed to fork and submit a PR.
