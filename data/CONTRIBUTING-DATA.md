# Contributing to the Commonweave Directory

> How to add, verify, or correct organizations in the directory.
>
> Read [`../AGENTS.md`](../AGENTS.md) first if you are an AI-assisted contributor.

The directory is the concrete deliverable of Commonweave. Its trustworthiness depends on every entry being sourced, classified honestly, and not exposing vulnerable groups.

This guide explains the schema, the safety rules, and the contribution workflow.

---

## Where the data lives

- **Canonical store:** `data/commonweave_directory.db` (sqlite). All ingest scripts read/write here.
- **Browsable index:** `data/search/<country>.json`, regenerated by `data/build_search_index.py`. This is what `directory.html` reads.
- **Map points:** generated by `data/build_map_points.py` and `data/build_map_v2.py`.

For most contributions you will edit either:

1. `data/search/<country>.json` directly (for a small batch of corrections), or
2. The DB via a script under `data/ingest_*.py` (for new sources).

For new sources, prefer adding an ingest script over editing the DB by hand. Scripts are reproducible.

---

## Tier model

Every record has a tier. Tiers are visible in directory views and weight downstream tooling.

| Tier | Meaning | Source examples |
|------|---------|-----------------|
| **A** | Hand-reviewed, gold standard | Manual curation, verified by Simon or a trusted reviewer |
| **B** | Registry-backed | UK Charity Commission, IRS BMF, Wikidata, ProPublica, OFN, ITUC |
| **C** | (reserved) | (not currently used) |
| **D** | Candidate | Web research, scraping, untriaged ingest |

A new ingest script should default to Tier D unless the source is a recognized registry.

---

## Legibility (added 2026-04-22)

Every record has a `legibility` field describing **how we found it**. This is critical for surfacing the directory's bias.

| Value | Meaning |
|-------|---------|
| `formal` | Found via state registry, charity commission, IRS, etc. |
| `hybrid` | Found via a mix of formal and community signals |
| `informal` | Found via mutual aid networks, community lists, social channels |
| `unknown` | Default until classified |

**All 26,022 current records read `unknown`.** Backfilling this is one of the highest-value contributions an agent can make (see `AGENT-TASKS.json` task `cw-legibility-backfill-001`).

## Schema (high-level)

Key columns in the `organizations` table:

```
id                  primary key
name                organization name
country             ISO 3166-1 alpha-2
city                city or admin region
description         short description
website             organization homepage
source              ingest source label (e.g. 'IRS_BMF', 'UK_CHARITY', 'WIKIDATA', 'WEB_RESEARCH', 'MANUAL')
source_url          link to the source record
tier                A / B / D
legibility          formal / hybrid / informal / unknown
alignment_score     integer, multi-pass keyword scoring vs framework mechanisms
framework_areas     json array of matched areas (housing, healthcare, food, etc.)
latitude, longitude (geocoded; nullable)
merged_into         id of canonical record if this row was merged via dedup_merge.py
created_at          ISO 8601
updated_at          ISO 8601
```

For the full schema, run `python data/inspect_db.py` or check the migration files in `data/migrations/`.

## Adding a new organization

### Required fields

- `name`
- `country`
- `description` (1-3 sentences)
- `source` and `source_url` (this is **required** — never add an unsourced org)
- `tier` (A or B; D if you can't verify)
- `legibility` (default `unknown` if uncertain)

### Optional but valued

- `website`
- `city`
- `framework_areas` (one or more of: housing, healthcare, education, food, ecology, democracy, conflict-resolution, cooperatives, recreation-arts, energy-digital)
- Geocoded `latitude, longitude` — but only if you actually verified the location

### Process

If you're adding a single org:

1. Edit `data/search/<country>.json`. Add the entry following the existing format.
2. In your PR, include the source URL and a one-line note at the top of the JSON file explaining what changed.
3. Don't regenerate the search index unless you're confident about all your changes — let the maintainer regenerate.

If you're adding many (>10) orgs from a new source:

1. Create an ingest script under `data/ingest_<source>.py`. Follow the pattern of `data/ingest_wikidata_bulk.py` or `data/ingest_ofn.py`.
2. Run it against a copy of the DB, inspect the diff, and submit the script + the diff in a PR.
3. Don't commit the modified DB — submit the script and let the maintainer re-run it.

## Correcting an existing organization

Common cases:

- **Broken website:** Edit the `website` field. If you can't find a working URL, leave it blank rather than inserting a tracking-redirect or stale archive link.
- **Wrong country:** Edit the `country` field. Note the correction in your PR.
- **Wrong framework area:** Edit `framework_areas`. Cite which mechanism the org is actually doing.
- **Misclassified as commons-aligned:** If the org is greenwashing or doesn't fit the framework, set `tier=D` and leave a note. Don't delete the row — the audit history is part of the project's credibility.

## Safety rule for vulnerable groups

This is the rule that overrides everything else:

> **Some organizations should not be made more visible without care.**

Examples of groups that need conservative handling:

- Mutual aid networks operating in repressive contexts.
- Migrant support, especially in countries with active deportation enforcement.
- Survivor-support organizations.
- Labor organizers in anti-union jurisdictions.
- Abolitionist work in carceral states.
- Reproductive health support in restrictive jurisdictions.
- Informal religious or cultural minority groups under persecution.

For these groups:

- Default `legibility=unknown`.
- Use country-level location only — no street addresses or building details.
- Prefer leaving a record incomplete over completing it speculatively.
- If the group has a public site that explicitly invites visibility, that consent is OK to follow.
- If you're unsure, **don't add the org**. The directory is improved more by 5 honest records than 50 reckless ones.

When you add or correct a record involving a sensitive group, note the safety reasoning in the PR.

## Audit and dedup

Two scripts run periodically to catch errors:

- **`data/dedup_merge.py`** — conservatively merges duplicates. Requires name + country + matching location signal (city, state, or lat/lng within 5km). Anything ambiguous goes to a human review file. Don't bypass this with manual deletes.
- **`data/staleness_check.py`** — flags records where the website has been broken across multiple checks. **Never auto-archives.** Flags for human review only. Skips informal/hybrid/no-website orgs by design.
- **`data/pipeline_auditor.py`** — weekly read-only self-improvement loop that writes evidence-cited proposals to `data/trim_audit/proposals-YYYY-MM-DD.md`.

## Validation checklist

Before submitting a directory PR:

- [ ] Source URL exists and resolves.
- [ ] Country code is ISO 3166-1 alpha-2.
- [ ] Description is specific (not boilerplate).
- [ ] If the org is sensitive, `legibility` is honest and detail is minimal.
- [ ] No invented citations.
- [ ] PR explains what was checked and how.

---

## Per-country JSON note format

When you edit `data/search/<country>.json`, add a top-level note key (or update the existing one) so the next contributor sees what changed:

```json
{
  "country": "ZA",
  "_notes": [
    {
      "date": "2026-04-25",
      "by": "your-handle (or 'Claude Sonnet 4.6 + your-handle')",
      "summary": "Verified 12 orgs; corrected 3 websites; flagged 1 potential greenwash for review."
    }
  ],
  "organizations": [
    ...
  ]
}
```

If `_notes` doesn't exist in a country file yet, you can be the contributor who adds it.
